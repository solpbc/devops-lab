## Session 2 — container build and end-to-end attestation

### 8. Repo hygiene
Added a `.gitignore` (macOS cruft, editor files, Rust/Python build artifacts, and attestation secrets like `*.pem`/`*.key`/`*.bin`/HCL blobs so they can never be committed). Went public on GitHub at `solpbc/devops-lab`.

### 9. Populated the `Containerfile`
Two-stage build on `ubuntu:24.04`: a builder stage compiles `snpguest` from source with `cargo build --release --features hyperv` (the `hyperv` feature is required on Azure — it enables the `--platform` path that reads the report from the vTPM, since `/dev/sev-guest` is hidden), and a slim runtime stage with `tpm2-tools`, `openssl`, `xxd`, `jq`. Documented that the Azure CVM customizations live in the **host VM image** (`Canonical:ubuntu-24_04-lts:cvm:latest`), not the container — marketplace images are VHDs, not OCI bases.

### 10. Added a test harness
`test/build-check.sh` runs three tiers without SEV-SNP hardware: `lint` (hadolint, warnings non-fatal), `build`, and `smoke` (`snpguest --help`). Fixed a `pipefail` lint warning in the Containerfile.

### 11. Built and ran on a real Azure CVM
Provisioned a `Standard_DC2as_v5` Confidential VM and built the image as `azureuser` (rootless podman). Key gotchas learned:
- `snpguest ok` **fails** on Azure ("Error reading MSR") — expected, because Azure hides the native SEV-SNP guest interface. Not a real failure.
- `snpguest report` needs the **raw** `/dev/tpm0`, not just `/dev/tpmrm0`.
- `/dev/tpm0` is owned `tss:root`; being in the `tss` *group* doesn't help until its group is changed. Workaround: `sudo chgrp tss /dev/tpm0 && sudo chmod g+rw /dev/tpm0`, plus `--group-add keep-groups` so rootless podman carries the membership in.
- `sudo podman` can't see the rootless-built image (separate storage) — so we stayed rootless.

### 12. `run.sh` now demonstrates the full AMD chain
Rewrote `run.sh` into a four-stage demo: (1) fetch report from vTPM via `--platform`, (2) decode it, (3) fetch the AMD cert chain (CA + VCEK) from the KDS, (4) verify the cert chain and report signature. Processor model / chip / TCB are all derived from the V3 report, so nothing is hardcoded. Degrades gracefully with a clear message when run off-hardware.

### 13. Verified a genuine report end-to-end ✅
On the CVM the full chain passes: ARK self-signed → ASK signed by ARK → VCEK signed by ASK; all four Reported-TCB values match; **VCEK signed the attestation report**. Confirmed genuine AMD-hardware-signed evidence with **no MAA in the trust path** — the repo's core thesis, demonstrated.
