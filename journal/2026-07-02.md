# Journal — 2026-07-02

## Hosted "solstone" service — exploration kickoff

Reviewed seven research docs (GPU CC threat model v1+v2, colocated VM infra guide, sub-$80
GPU node reality check, H100 CC fleet build guide, RoCE v2 cost analysis, multi-node
form factors) as the basis for a hosted confidential-compute service. Settled the
framing in discussion:

- **Phasing:** P1 = cheap SEV-SNP VMs (~$3/VM, R6515 dense-rack model), fully
  operator-untrusted. P2 = H100-PCIe CC-mode GPU pool (separate pool, one GPU per
  CVM, attested-TLS bridge), gated on demand (~$460–610K for 8 nodes). Optional
  P1.5 = CPU-confidential + ordinary low-end GPU inference.
- **Trust anchor:** operator-untrusted. The trust inversion from the repo's current
  work: solpbc today is us-verifying-Azure; solstone is tenants-verifying-us, with
  the operator as the Tier 1 adversary SEV-SNP defeats. On our own bare metal the
  chain is simpler than Azure (native /dev/sev-guest, real launch measurements, no
  HCL/vTPM indirection) — verifier.py largely carries over minus the Azure parsing.
- **Attestation bridge is the core IP** across phases: key release gated on
  tenant-side verification, no trust in the operator or any Microsoft/MAA-style
  intermediary.
- P2 residual: bounce-buffer CC leaves the PCIe-tap gap (Tier 2), and the party
  attesting physical controls is the operator tenants aren't supposed to trust.
  Mitigate via third-party-audited facility controls now; TDISP (~2027) removes it
  cryptographically.

## Two paths flagged for study

### 1. Reference-values strategy (P1 blocker)

The repo's record-then-pin PCR policy means "trust the first boot" — fine for a
research spike, circular for a hosted product. Tenants need a way to compute
expected launch measurements independently of us. Options to evaluate:

- **Reproducible measured guest images:** we publish image build pipeline +
  expected SEV-SNP launch digest; anyone can rebuild and confirm.
- **Tenant-supplied images:** tenant brings their own image and computes their own
  measurement; strongest trust story, most ops friction (patching, support).
- **Published-measurement registry:** signed registry of known-good measurements
  per image/firmware version; simplest UX, weakest independence (registry is ours —
  needs transparency-log or third-party countersigning to avoid circularity).

This decision back-pressures everything: guest image pipeline, host patching
cadence (TCB changes invalidate pinned values), migration policy, SLA. Study
question: which option (or hybrid) makes "operator-untrusted" verifiable rather
than asserted, at acceptable ops cost?

### 2. Does P1.5 (low-end GPU inference tier) belong?

The threat-model v2 doc adds SEV-SNP + non-CC GPU as a first-class configuration
and settles the technical facts: non-CC GPU VRAM and CPU↔GPU DMA are *trivially
readable* by the Tier 1 adversary (the operator — us), DMA remapping is only
partially mitigated (GPU-side targets redirectable), and — the deeper product
problem — non-CC GPUs have **no attestation capability at all**: no Secure
Processor, no way for a tenant to verify GPU identity, configuration, or even
which card they got. P1.5 wouldn't carry a caveat; it sits entirely outside the
attestation-verified story that defines the service.

v2's gating question is the right frame: "does the GPU process confidential
data?" If no (tenant explicitly runs non-sensitive GPU work; CPU side stays
confidential), SEV-SNP-only is a legitimate labeled tier. If yes, CC-mode GPU
(P2) or nothing. Study question narrows to: is there demand for a
"GPU-non-confidential by declaration" tier, and can the tenant-facing contract
make the boundary unmistakable (e.g. attestation report that covers CPU only,
GPU explicitly flagged unverified)?

## Later: separated analyses; Azure confidential-child-capable VMs

### Azure EC*_cc_v5 / DC*_cc_v5 (preview): nested path to the same product

Evaluated Azure's confidential-child-capable series (ECas_cc_v5/ECads_cc_v5,
D-series siblings; preview, AKS-gated). The parent VM is deliberately
*non-confidential*; it exposes nested SEV-SNP so the renter creates
confidential **child** VMs. Unlike DCas_v5/ECas_v5 (where the Azure VM is the
CVM and anyone inside trusts whoever holds it), this is the only Azure SKU
where solstone's trust shape is possible: we operate the parent as hypervisor,
tenants get SNP children protected from both us and Azure's host.

- **Cost:** EC96as parent (96 vCPU/672GiB, ~$2.5K+/mo on-demand, ~40% off
  3-yr reserved) → ~37–40 16GB children → ~$60–120/child/mo on-demand,
  ~$40–70 reserved. Owning still wins 13–40x; core on-prem conclusion stands.
- **What it adds:** a zero-capex pilot/burst tier — validate demand and the
  tenant-verifier UX before committing rack capital. Also a trust wrinkle:
  we never touch physical hardware, so the Tier 2 physical adversary becomes
  Microsoft's audited datacenter rather than our self-attested colo cage.
- **E vs D series:** product is RAM-bound (same reason DIMMs were the binding
  variable on-prem) → E-series at ~7GiB/vCPU beats D at ~4GiB/vCPU on
  $/child.

### Bare metal vs nested: the rest of the delta

- **Child TCB/firmware control (biggest open question):** children launch
  through Azure's stack, which likely inserts their paravisor/firmware into
  the child's measured TCB — possibly no custom OVMF, possibly no ID block.
  If children get an HCL-mediated report instead of native /dev/sev-guest,
  the Azure verifier machinery we planned to retire comes back. Verify first.
- **TCB cadence:** reported TCB values move on Microsoft's host-patch
  schedule, not ours; tenant policies must track a third party.
- **Failure domains:** no live migration / memory-preserving updates — parent
  maintenance reboots ~40 tenants on Azure's schedule; on our racks we own
  the window.
- **Performance:** nested adds a second virtualization layer; children get
  software I/O through the parent (no SR-IOV into a child). The 4:1
  overcommit behind $3.07/VM compounds badly on already-virtualized vCPUs.
- **Network/egress:** children NAT behind the parent NIC; Azure egress
  (~$0.05+/GB) vs colo unmetered can dominate for upload-heavy tenants.
- **Co-tenancy:** parent shares physical hosts with arbitrary Azure
  customers; on our racks, co-tenants are only our own tenants.
- **Ceiling:** no GPU in the child-capable model (P2 is bare-metal or Azure
  NCCads regardless); preview terms (no SLA, withdrawable) cap it at pilot.

**Position:** nested is attractive exactly once — as a demand-validation
pilot where zero capex and Microsoft-held physical custody matter most. The
product's long-term shape still points at bare metal.
