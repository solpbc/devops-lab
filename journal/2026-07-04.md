# Journal — 2026-07-04

## AKS EC*_cc testbed: built it, then found the door welded shut

Wrote `aks-cc-testbed.sh` (repo root, committed d5e9313) plus `yaml/snp-probe.yaml`
and `yaml/parent-probe.yaml` to answer the 07-02 verification task: child-VM
attestation path on EC*_cc_v5 (native /dev/sev-guest vs HCL-mediated; firmware and
ID-block control). Two-layer design: a kata-cc pod probing inside the child UVM
(report source, HOST_DATA vs CCE policy hash, CHIP_ID, THIM endpoint), and a
privileged runc pod probing the parent (/dev/sev, kvm_amd sev_snp, what launches
the UVMs). CCE policy generated via `katapolicygen --print-policy` and injected
client-side at apply time so the manifest stays pristine; policy SHA256 recorded as
the expected HOST_DATA value.

### Provisioning lessons (mundane but time-consuming)

- **CSE `VMExtensionError_K8SDownloadTimeout` with curl 404 is not a firewall
  problem** — the boilerplate error text misdirects. 404 means the node reached the
  endpoint and the artifact doesn't exist: stale k8s-version/node-image combo.
  Azure Linux 2.0 node images were removed 2026-03-31 and old images referenced the
  retired acs-mirror.azureedge.net CDN. Fix: pin a current `--kubernetes-version`
  (used 1.35.5), keep aks-preview extension updated, and create the system pool and
  the special pool separately so failures isolate.
- **eastus SKU churn:** D4as_v5 and D4as_v6 both unavailable to this subscription;
  D4s_v7 (Intel Xeon 6 Granite Rapids) worked. Naming decoder: `a` = AMD, no-`a` =
  Intel; `s` = premium storage, `d` = local disk, `l` = low memory. For a system
  pool none of it matters. Useful query for what's actually deployable:
  `az vm list-skus -l $LOC --resource-type virtualMachines --all --query
  "[?starts_with(name,'Standard_D4') && length(restrictions)==\`0\`].name"`.
- **Quota was not the blocker:** cc families have their own buckets
  (standardECASccv5Family etc.); this subscription has 65 vCPUs per cc class.
- Sandbox note: git operations from the Cowork session leave `.git/*.lock` and
  `tmp_obj_*` litter it can't unlink; clean with
  `rm -f .git/*.lock .git/objects/*/tmp_obj_*` before the next git write.

### The real finding: Confidential Containers preview sunset March 2026

The EC8as_cc_v5 node pool failure is not fixable. Microsoft sunset the AKS
Confidential Containers preview in **March 2026**: existing KataCcIsolation node
pools keep running, **new nodes cannot be created**. The SKU still shows available
(hardware exists, quota granted) but the only supported consumer of it is switched
off. No k8s version dodges this; docs never had an upper version bound — the gate
is the sunset.

Consequences for the 07-02 analysis:

- The "zero-capex pilot/burst tier" via AKS kata-cc **no longer exists**. The
  caveat we logged ("preview terms cap it at pilot — no SLA, withdrawable") wasn't
  a caveat, it was the outcome. Strengthens the bare-metal conclusion: Azure
  withdrew the only SKU family matching solstone's trust shape.
- The cc_v5 series is now orphaned hardware from our perspective — parent-hosted
  SNP children with us as hypervisor has no supported managed path on Azure.
- Microsoft's pointed replacements: CVM node pools on AKS (DCas_v5 model — the
  us-verifying-Azure shape solpbc already covers, not nested), **Confidential
  Containers on ACI (GA)**, ARO Confidential Containers, or open-source CoCo on
  our own hardware.

## Later: Layer-2 resolved empirically — the parent was never a hypervisor

Ran the plain-nodepool probe (`ccplain` pool, EC8as_cc_v5, no `--workload-runtime`;
pool provisioned fine — the sunset gate applies to the KataCcIsolation runtime, not
the SKU). Privileged probe results:

- `svm` present in cpuinfo and `kvm_amd` loads: generic nested virtualization is
  real; ordinary (non-confidential) nested VMs would work.
- **No `/dev/sev`**: no PSP/ccp exposure to the parent, so KVM cannot launch
  SEV-SNP children. The `sev*` kvm_amd parameters are off.
- **No `/dev/mshv`, no mshv module, no Hyper-V root-partition signs**: Azure's
  actual child-launch path (node booted as L1 root partition under the Microsoft
  hypervisor, children created via mshv + cloud-hypervisor + IGVM) is baked into
  the special CC node image at provisioning time and is absent on a stock node.

Resolution of 07-02's "biggest open question": on EC*_cc_v5 the SNP-child
capability was never a SKU property the renter could drive. There is no PSP and no
reachable launch interface; children were only ever creatable through Microsoft's
L0-provisioned MSHV stack via the (now sunset) KataCcIsolation node image. "We
operate the parent as hypervisor" was never actually on offer — the parent was a
delegated launcher inside Microsoft's stack. The nested pilot tier isn't just
withdrawn; the trust shape it promised didn't exist. Bare metal stands.

Kata-cc pool: failed after hours in `Creating`, with the same
`VMExtensionError_K8SDownloadTimeout` / curl 404 as the original cluster create.
Diagnosis is airtight now: the ccplain pool succeeded on the same K8s version
(1.35.5), region, and SKU — the only delta is the KataCcIsolation node image. The
sunset isn't enforced by a policy gate; it's enforced by deleted artifacts (the CC
node image lineage was AL2-based, AL2 images were removed 2026-03-31, and the CC
image was never rebuilt on AL3). Layer 1 on AKS is dead; ACI CoCo is the remaining
(GA) venue for the attestation-shape questions.

## Later still: Layer 1 answered on ACI — native path, no HCL, no MAA

Ported the probe to ACI Confidential Containers (aci-cc-testbed.sh +
arm/aci-snp-probe.json). Getting there was its own lesson chain: confcom
acipolicygen is Linux/Windows-only (ran it in an azure-cli container via podman
with `--tar` for daemon-free layer hashing); image strings must match tar
RepoTags byte-for-byte (podman fully qualifies names); Docker Hub throttles
anonymous pulls from ACI IP ranges (switched to mcr.microsoft.com/mirror/...);
and podman on Apple Silicon pulls arm64 by default while policies are amd64
(`--platform linux/amd64`, else confcom crashes in its own error path).

Probe results, all green — this is the shape we wanted:

- **`/dev/sev-guest` present, no vTPM, no /dev/sev.** Raw SNP guest interface.
- **Report at VMPL 0, version 3**: the UVM runs unparavisored — the entire
  HCL/vTPM indirection solpbc handles on Azure CVMs does not exist here.
- **Nonce echoed in REPORT_DATA**: freshness binding via the ioctl works as on
  bare metal.
- **HOST_DATA == sha256 of the CCE policy** (34ba5e28…bbd7): the policy the
  tenant authored is hardware-bound in the report, verifiable by anyone —
  MAA nowhere in the path.
- **CHIP_ID real**: VCEK fetchable directly from AMD KDS with reported_tcb
  SPLs; THIM (169.254.169.254/metadata/THIM/amd/certification) as the faster
  in-fabric alternative. Trust roots stay AMD-only (roots/amd/ applies).
- **id_key_digest nonzero**: UVM launched with a Microsoft ID block;
  MEASUREMENT should match reference values published in
  microsoft/confidential-sidecar-containers.

Conclusion: the tenant-verifier UX solstone needs (raw report, policy hash in
HOST_DATA, AMD-rooted chain, no platform-attestation intermediary) is exactly
what ACI CoCo child UVMs expose. verifier.py applies nearly as-is — the Azure
CVM parsing (lib/hcl.sh, NV-index reads) is simply unnecessary on this path.
Remaining to close: VCEK fetch + signature-chain appraisal offline against
report.bin, and MEASUREMENT vs Microsoft's published UVM reference values.
Teardown: az group delete -n solpbc-acicc-rg.
